# blog-butterfly-go 后台内容创作工作台 P2 PRD

> 版本：v1.0  
> 状态：已实现（基于当前代码与交互现状整理）  
> 面向模块：`/root/blog-butterfly-go/admin-vue`、`/root/blog-butterfly-go/backend`  
> 关联主题：内容创作效率增强（Markdown 编辑、图床联动、自动保存草稿）

---

## 1. 背景与现状

### 1.1 当前项目现状
`blog-butterfly-go` 已具备以下后台基础能力：

- 文章列表、筛选、分页、编辑、新建、删除
- 分类管理
- 图床上传、列表展示、复制链接、批量删除
- 文章字段支持：标题、摘要、封面、分类、标签、置顶、状态、正文
- 后端已支持文章状态 `draft/published`
- 管理后台主线为：
  - `admin-vue/src/`
  - `admin-vue/src/views/`
  - `admin-vue/src/components/`

### 1.2 已确认的当前实现事实
基于代码审查，当前现状如下：

1. **后台主线已切换到 `admin-vue`**
   - 当前管理端以 Vue 3 + Vite 为实现基础
   - 文章创建、编辑、分类、素材等工作流均应继续围绕 `admin-vue` 演进
2. **Markdown / 创作效率能力仍是产品重点**
   - 编辑体验、预览、快捷保存、图床联动、自动保存等方向仍然成立
   - 但落点应基于 `admin-vue` 当前实现继续建设，而不是回到旧原生后台
3. **旧版原生后台已转入归档**
   - 历史目录已从 `frontend/` 迁移为 `frontend-已归档/`
   - 仅作为历史参考，不再作为主线实现位置
4. **图床与文章编辑仍需保持联动**
   - 支持上传、加载图片列表、复制链接、批量删除
   - 支持封面选择与正文插图等高频创作场景
5. **草稿保护仍是关键体验**
   - 手动保存与自动保存的状态表达需要持续清晰
   - 应继续避免用户误以为本地自动保存已同步到服务端

### 1.3 当前问题
当前后台虽然“能发文章”，但距离“高效创作工具”仍有明显差距：

- 正文编辑体验偏原始，缺少 Markdown 预览与编辑增强
- 图片插入流程割裂，图床与正文编辑没有打通
- 用户输入内容存在丢失风险，缺少自动保存与恢复机制
- 归档实现与 `admin-vue` 主线之间需要明确边界，避免后续误改到历史代码

## 2. 产品定位

将后台从“文章管理面板”升级为“内容创作工作台”。

核心目标不是增加后台菜单，而是提升作者在后台中完成一篇文章的效率、流畅度与安全感。

### 2.1 目标关键词

- 写得顺：Markdown 编辑器 + 预览
- 配图快：图床一键插入封面 / 正文
- 不怕丢：自动保存草稿

### 2.2 产品价值

对内容作者而言，这三个能力属于高频核心链路：

1. 输入正文
2. 插入图片
3. 保存与恢复内容

一旦这条链路顺滑，后台才真正具备“可持续写作”的产品价值。

---

## 3. 目标用户

### 3.1 主要用户
博客管理员 / 内容作者（当前即站长本人）

### 3.2 用户特征

- 高频进行 Markdown 文章创作
- 会频繁配封面图与正文配图
- 更关注创作效率，而非复杂协作流
- 对“内容丢失”容忍度极低

---

## 4. 用户场景

### 场景 A：新建文章
用户点击“新建文章”，默认进入草稿编辑状态，可直接输入标题、摘要、正文，并快速选择封面图。

### 场景 B：正文插图
用户在 Markdown 正文中写到某一段时，需要从已上传图床中选择图片，并将 Markdown 图片语法插入到当前光标位置。

### 场景 C：意外关闭 / 刷新
用户正在写文章时刷新页面、误关闭页面、浏览器崩溃，重新进入编辑器后应能恢复本地自动保存内容。

### 场景 D：编辑已有文章
用户打开已有文章修改内容，系统应区分：
- 服务器已保存的正式内容
- 浏览器本地尚未提交的自动保存草稿

---

## 5. 范围定义

## 5.1 本期范围（P2）

### P2-1 Markdown 编辑器 + 预览
在当前后台 modal 编辑器中引入 Markdown 编辑体验，并提供预览能力。

### P2-2 图床一键插入封面 / 正文
将图床能力嵌入编辑器流程，支持从图片库中直接选择并回填。

### P2-3 自动保存草稿
基于本地存储实现自动保存、恢复、状态提示、草稿清理。

## 5.2 非本期范围
以下能力不纳入本期：

- 多人协作编辑
- 服务端草稿版本历史
- 评论区审核流
- 图床高级素材管理（标签、相册、搜索）
- 所见即所得富文本编辑器替代 Markdown

## 6. 信息架构与交互原则

### 6.1 后台结构原则
现役主后台继续以 `admin-vue` 为单一主线；归档目录 `frontend-已归档` 仅保留历史参考价值。

### 6.2 编辑器原则
编辑器是后台核心工作区，应遵循：

- 单入口：所有新建 / 编辑都走同一套 modal 编辑器逻辑
- 单状态源：表单数据统一由主编辑器状态管理
- 单图床入口：封面选择与正文插图复用同一个图片选择面板

### 6.3 文案原则
明确区分：

- `已自动保存到本地`
- `已保存到服务器`

避免用户误认为自动保存已同步到服务端。

## 7. 功能需求

## 7.1 P2-1 Markdown 编辑器 + 预览

### 7.1.1 功能描述
在当前文章编辑弹窗中，将正文编辑区域从普通 textarea 升级为 Markdown 编辑器。

### 7.1.2 具体需求

1. 支持 Markdown 语法编辑
2. 支持预览模式
3. 桌面端优先支持“双栏编辑 / 预览”或“编辑 / 预览切换”
4. 至少保留以下常用编辑能力：
   - 标题
   - 粗体 / 斜体
   - 引用
   - 列表
   - 代码块
   - 链接
   - 图片
5. 支持快捷键保存：
   - Windows/Linux：`Ctrl + S`
   - macOS：`Cmd + S`
6. 编辑器中的图片按钮需可联动图床选择器
7. 编辑器内容变更应触发自动保存机制

### 7.1.3 推荐方案
优先复用当前 `admin-vue` 中已验证的编辑器与内容工作流经验，避免无必要的大规模重构成本。

### 7.1.4 验收标准

- 打开文章编辑器后，正文不再是原始 textarea
- 用户可进行 Markdown 编辑与预览
- 快捷键保存可触发保存动作
- 编辑器中插图动作可连接图片选择流程

---

## 7.2 P2-2 图床一键插入封面 / 正文

### 7.2.1 功能描述
提供统一图片选择器，将图床从独立管理页扩展为编辑器素材来源。

### 7.2.2 封面插入需求

1. 在封面区域提供“从图床选择”按钮
2. 点击后弹出图片选择器
3. 用户点击某张图后：
   - 自动回填封面 URL
   - 自动刷新封面预览
4. 保留已有“换一张随机封面”能力

### 7.2.3 正文插图需求

1. 在 Markdown 编辑器工具栏提供“插入图片”入口
2. 点击后弹出统一图片选择器
3. 用户选中图片后，在当前光标位置插入 Markdown 图片语法：

```md
![图片描述](图片URL)
```

4. 插入位置应为当前光标位置，而不是固定追加到末尾
5. 可选提供辅助动作：
   - 复制 URL
   - 插入纯链接

### 7.2.4 验收标准

- 选择封面时无需手动切换到图床页复制 URL
- 正文插图时无需手动拼 Markdown 语法
- 同一套图片选择面板可复用于封面与正文两类场景

---

## 7.3 P2-3 自动保存草稿

### 7.3.1 功能描述
当用户在编辑文章时，系统自动将当前编辑状态保存到浏览器本地，以防内容丢失。

### 7.3.2 保存策略

1. 当以下字段发生变化时触发自动保存：
   - 标题
   - 摘要
   - 封面
   - 分类
   - 标签
   - 状态
   - 置顶
   - 正文
2. 使用防抖策略，推荐 2~5 秒
3. 草稿存储至 `localStorage`
4. key 规则建议：
   - 新建文章：`admin:draft:new`
   - 编辑文章：`admin:draft:article:<id>`

### 7.3.3 恢复策略

1. 打开编辑器时检测本地草稿
2. 若存在未清理草稿，则提示用户：
   - 恢复本地草稿
   - 忽略本地草稿
3. 对已有文章编辑场景，应避免无提示覆盖服务端内容

### 7.3.4 状态提示
页面需展示明确的保存状态，例如：

- 未保存
- 正在自动保存到本地...
- 已自动保存到本地 14:32:18
- 自动保存失败
- 已保存到服务器

### 7.3.5 清理策略
以下场景清理本地草稿：

1. 手动保存成功
2. 文章发布成功
3. 用户主动放弃草稿

### 7.3.6 验收标准

- 用户修改任意内容后，无需手动操作也能完成本地草稿保存
- 刷新 / 关闭后可恢复未提交内容
- 本地自动保存与服务器保存有清晰区分

## 8. 交互细节要求

### 8.1 编辑器状态栏
编辑器区域建议增加状态栏，展示：

- 保存状态
- 当前文章模式（新建 / 编辑）
- 草稿 / 已发布状态
- 字数统计（可选）
- 阅读时长估算（可选）

### 8.2 图片选择器
统一弹层应包含：

- 图片缩略图
- URL 展示
- 当前操作模式提示（选择封面 / 插入正文）
- 取消按钮

### 8.3 关闭编辑器行为
当前实现中，若存在未提交且未同步到服务器的改动，关闭编辑器时会弹出确认提示：

- 继续关闭：保留本地草稿，稍后重新打开时可恢复
- 取消关闭：继续留在编辑器中修改

补充说明：
- 当前关闭弹窗内**没有**单独的“保存后关闭”二次操作按钮
- 若存在待触发的自动保存定时器，关闭时会先落一次本地草稿，再关闭编辑器

## 9. 成功指标

### 9.1 体验指标

- 创建一篇文章的操作步骤减少
- 插图路径缩短
- 用户无需频繁手动复制图床 URL
- 误关闭 / 刷新后内容恢复成功率提升

### 9.2 可观察信号

- 编辑器中“插图”使用频率提升
- 封面图选择成功率提升
- 因内容丢失导致的重复输入显著减少

## 10. 技术约束与风险

### 10.1 约束

1. 当前后台主流程为单页 + 弹窗模式，不宜本期大改为多页复杂路由
2. 现有 `editor.html` 为旧实现，不应再继续扩展为主链路
3. 当前后端未提供服务端自动草稿接口，本期应优先使用本地草稿方案
4. 当前图床接口已可满足图片选择能力，无需本期新增后端图片查询结构

### 10.2 风险

1. **双实现分裂风险**
   - 若继续同时维护 `editor.html` 与 `index.html` 编辑流，会造成能力重复与状态不一致
2. **自动保存认知风险**
   - 若文案不清晰，用户会误以为自动保存等于服务端已保存
3. **插图光标风险**
   - 若图片插入不能落到当前光标位置，体验会明显下降
4. **模态框容量风险**
   - 当前 modal 编辑器如继续堆叠功能，可能需要重新组织布局与滚动体验

---

## 11. 优先级建议

### 第一优先级：持续打磨自动保存草稿体验
原因：基础能力已落地，后续更值得优化的是恢复提示、边界场景与稳定性体验

### 第二优先级：持续打磨图床一键插入封面 / 正文
原因：主链路已打通，后续可围绕素材检索、插入效率和交互细节继续增强

### 第三优先级：持续优化 Markdown 编辑器与工作区布局
原因：随着内容创作能力继续增长，仍可能需要进一步梳理 `admin-vue` 的编辑布局与交互体验

---

## 12. 最终结论

本期 P2 不应继续偏向“后台管理功能堆叠”，而应聚焦“创作链路增强”。

三项能力组合后的结果是：

- Markdown 编辑器 + 预览：解决“写得顺”
- 图床一键插入：解决“配图快”
- 自动保存草稿：解决“内容不丢”

这会让 `blog-butterfly-go` 后台从“能发布文章”升级为“愿意长期使用来写文章的创作工具”。
